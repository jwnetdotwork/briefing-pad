# ブリーフィング・パッド / briefing-pad 仕様ドラフト

## 1. アプリ概要

### アプリ名

**ブリーフィング・パッド**  
英語名 / プロジェクト名：`briefing-pad`

### 目的

2〜5分程度の実演を聞きながら、リアルタイム文字起こしとLLM分析を使って、短評に使えそうな観察材料を整理する。

短評文を完全自動生成することが主目的ではなく、短評者がその場で判断しやすくするための **補助ツール** とする。

---

## 2. 基本方針

### アプリの位置づけ

- アプリは評価者ではなく **補助者**
- 最終判断はユーザーが行う
- 実演者への評価をAIに任せるものではない
- 実演中・実演直後に、短評で使えそうな材料を素早く提示する

### MVPでやること

- リアルタイム文字起こしを表示
- 事前に定義されたパート情報を読み込む
- 現在のパートに基づいてLLM分析する
- 観察メモに該当しそうな内容を表示
- 良かった点に該当しそうな内容を表示
- 低信頼候補は表示しない
- パート終了後、NotionにAIメモを書き込む

### MVPでやらないこと

- 実演中の細かい採用/却下操作
- ユーザーによるその場での詳細編集
- 既存Notionメモの直接上書き
- Notionの既存観察メモ欄の穴埋め
- 次の一歩の自動提案
- Notionページ全体の自由形式パース
- 完全な短評文の自動作成を主目的にすること

---

## 3. 対象プラットフォーム・技術

### プラットフォーム

- macOS 26+
- 日本語のみ対応

### UI

- Swift
- SwiftUI

### 音声入力

- AVFoundation

### リアルタイム文字起こし

- Apple SpeechAnalyzer
- Apple SpeechTranscriber

### LLM

- OpenAI API

### 外部連携

- Notion API

---

## 4. 想定利用フロー

```text
1. ユーザーがNotionページを指定する
2. アプリがNotionページのブロック構造を読み込む
3. ルールベースでパート情報をパースする
4. ユーザーが現在のパートを選択する
5. マイク入力を開始する
6. リアルタイム文字起こしを表示する
7. 確定した文字起こしをチャンク化してLLMへ送る
8. LLMが観察メモ・良かった点の該当候補を返す
9. アプリ画面に高信頼候補だけを表示する
10. パート終了後、最終処理を行う
11. Notionの該当パート内の🤖AIメモブロック配下に結果を書き込む
12. ユーザーはiPadのNotionでAIメモを見て短評に使う
```

---

## 5. Notion連携仕様

## 5.1 Notionを使う目的

- 既存の短評用メモ環境としてNotionを利用する
- Macで実演を見ながら処理し、iPadのNotionで短評時に確認する
- コピペの手間を省く
- ただし、既存メモを壊さないことを最優先する

---

## 5.2 Notion読み込み

### 入力

- ユーザーがNotionページを指定する
- アプリはNotion APIで対象ページのブロック構造を取得する（レート制限回避のため1リクエストごとに0.33のwaitを入れる）

### パース方針

まずは **ルールベースでパース** する。

LLMによる自由パースはMVPでは前提にしない。

### 想定テンプレート

Notionページは以下のような構造を想定する。

```markdown
### 4. 会話を始める

（4分）日常生活で。

📓学習ポイント
- 愛込 レッスン3 ポイント5「助けになる」
- 相手の助けになりそうなことを積極的に行う

👀観察メモ
- 相手の困り事・気になった様子：
- 実際に差し伸べた助け：
- 助けた時のやわらかい一言：
- 相手が安心したと感じられる反応：

👍どこがどのように良かったか
- 親切をきっかけにしたので、最初から構えさせにくい流れ
- 助けたい気持ちが先に出ると、真心が伝わりやすい

🤖AIメモ
```

### パース対象

主に以下を抽出する。

- パート番号
- パートタイトル
- 所要時間
- 場面設定
- 学習ポイント
- 観察メモ項目
- 良かった点候補
- 🤖AIメモブロック
- 各ブロックID

---

## 5.3 Notion書き込み場所

アプリが書き込んでよい場所は、原則として：

```text
🤖AIメモブロック配下のみ
```

### 書き込まない場所

- 既存の学習ポイント
- 既存の観察メモ
- 既存の良かった点候補
- ユーザーが手で書いたメモ
- Notionページ内のその他の任意ブロック

---

## 5.4 AIメモが存在する場合

### アプリ作成のAIメモの場合

- 更新してよい

### ユーザーが編集していた場合

- 上書きしない
- 新しいAIメモとして追記する

### 判定方法

Notion読み込み時点の内容を保存しておき、書き込み前に再取得して比較する。

比較対象の候補：

- block ID
- last_edited_time
- plain text
- content hash

---

## 5.5 Notion書き込み内容

iPadで短評時に見るため、短く圧縮された内容を書き込む。

### 表示量

- 短評で使えそうな点：最大2件
- 根拠になりそうな観察：最大2件
- 短評サンプル：1件

### 書き込み例

```markdown
🤖 AIメモ 20:42更新

◎ 短評で使えそう
- 親切な行動が会話のきっかけになっていた。
- 相手を助けたい気持ちが先に出ていて、学習ポイントに合っていた。

👀 根拠になりそうな観察
- 「お手伝いしましょうか」と声をかけていた。
- 助けた後に自然に会話が広がった。

💡 言えそうな一言
親切な行動が先にあったので、相手も構えずに会話に入りやすかったと思います。
```

---

## 5.6 Notion書き込み失敗時

- アプリ内に結果は残す
- 自動リトライはしない
- 手動リトライ可能にする

---

## 6. パート情報仕様

## 6.1 基本方針

パート定義はガチガチに構造化しすぎない。  
ただし、LLM処理とUI表示に必要な最低限は構造化する。

方針：

```text
Notion上では柔軟に書ける
アプリ内では必要な部分だけゆるく構造化する
```

---

## 6.2 内部データ例

```json
{
  "partId": "part-4",
  "partNumber": 4,
  "title": "会話を始める",
  "durationMinutes": 4,
  "setting": "日常生活で",
  "rawMarkdown": "...",
  "learningPoints": [

```

## 7. リアルタイム文字起こし仕様

## 7.1 音声入力

- AVFoundationでマイク入力を取得する

## 7.2 文字起こし

- Apple SpeechAnalyzer + SpeechTranscriberを使用する
- 日本語のみ対応
- 確定した文字起こしのみLLMへ送信する

## 7.3 表示目標

文字起こしは、

```text
3秒以内程度の遅延
```

を目標とする。

---

## 8. チャンク化仕様

## 8.1 基本方針

確定した文字起こしをチャンク化してLLMへ送る。

### チャンク化条件

基本条件：

```text
無音が1秒程度あったらチャンク化
```

保険条件：

```text
15秒程度経過したらチャンク化
```

想定：

```text
無音1秒
または
15秒経過
または
一定文字数到達
```

---

## 8.2 LLMへ送る内容

毎回すべての全文を送る。

```text
これまでの確定文字起こし
今回の新しい確定文字起こし
すでに高信頼で検出済みの項目
```

### 注意

全トランスクリプトはアプリ内に保存するが、毎回LLMへ全文送信することは避ける方向。

---

## 9. LLM分析仕様

## 9.1 LLMの役割

LLMは、現在のパート情報と文字起こしをもとに、以下を判定する。

- 観察メモ項目に該当しそうな点
- 良かった点候補に該当しそうな点
- 短く表示できる根拠・理由

---

## 9.2 出力方針

- 低信頼候補は表示しない
- 高信頼のものを優先して画面表示する
- UI上では一目で分かるようにする
- 内部的には信頼度や状態を持ってよい

### 信頼度表示案

```text
◎ かなり使えそう
○ 候補として使えそう
△ 弱い候補
```

MVPでは：

```text
△以下、または低信頼候補は非表示
```

---

## 9.3 LLM出力形式

JSON固定にするかは未確定。  
ただし、UI反映の安定性を考えると、構造化出力が望ましい。

想定例：

```json
{
  "observationMatches": [
    {
      "itemId": "obs-4-01",
      "confidence": 0.86,
      "shortEvidence": "相手の困り事に気づいて声をかけていた"
    }
  ],
  "positiveMatches": [
    {
      "itemId": "pos-4-01",
      "confidence": 0.88,
      "shortReason": "親切な行動が会話の入口になっていた"
    }
  ]
}
```

## 10. UI仕様

## 10.1 基本方針

実演中に細かく操作する時間はないため、UIは **見るだけで分かる** ことを重視する。

### 優先順位

```text
速い
見やすい
邪魔しない
低信頼候補を出さない
あとから確認できる
```

---

## 10.2 表示内容

### 実演中

- 現在のパート
- リアルタイム文字起こし
- 観察メモ項目の状態
- 良かった点候補
- 信頼度アイコン
- 高信頼候補のみ

### パート終了後

- 短評で使えそうな点 最大2件
- 根拠になりそうな観察 最大2件
- 言えそうな一言 1件
- Notion書き込み状態

---

## 10.3 操作

MVPでは、実演中の採用/却下操作は行わない。

主な操作：

- Notionページ指定
- パート選択
- 録音開始
- 録音停止
- パート終了
- Notionへ書き込み
- 書き込み失敗時の手動リトライ

---

## 11. パート終了後処理

## 11.1 処理内容

パート終了時に、リアルタイム中の解析結果をもとに最終整理を行う。

- 高信頼候補の整理
- 重複候補の統合
- Notion書き込み用の短いまとめ作成
- 言えそうな一言の生成

---

## 11.2 時間目標

パート終了後処理は、

```text
5〜10秒以内
```

を目標とする。

ただし、Notion API書き込みを行うため、処理が多少遅れてもアプリ内には結果を保持する。

---

## 12. データ保存方針

## 12.1 保存するもの

以下は保存可能にする。

- 音声
- 文字起こし
- LLM出力
- 個人名を含むパート情報
- Notionページ情報
- NotionブロックID
- AIメモ書き込み履歴

---

## 12.2 削除

- セッション単位で削除可能にする
- 音声、文字起こし、LLM結果を後から削除できるようにする

---

## 13. ネットワーク障害時

## 13.1 OpenAI API失敗時

- 文字起こしは可能な範囲で継続
- LLM分析に失敗した場合は、アプリ内で失敗状態を保持
- 自動リトライは必須にしない

## 13.2 Notion API失敗時

- アプリ内に結果を残す
- ユーザーが手動でリトライする

---

## 14. セキュリティ・安全設計

## 14.1 Notion上書き防止

アプリは原則として、

```text
🤖AIメモブロック配下のみ
```

を操作する。

### 上書き防止ルール

```text
1. 既存の人間メモは上書きしない
2. アプリ作成のAIメモは更新可
3. ユーザーがAIメモを編集していた場合は上書きしない
4. その場合は新しいAIメモとして追記する
5. 書き込み前に対象ブロックを再取得して確認する
```
