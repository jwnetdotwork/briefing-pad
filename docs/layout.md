おすすめ画面構成はこんな感じ。

```text
┌──────────────────────────────┐
│ セッション選択 ▼   ＋   🗑     │
├──────────────────────────────┤
│ Part 4. 会話を始める           │
│ 持ち時間: 4分                  │
│                              │
│ [前へ]  00:02:31  [次へ]       │
│ [開始]           [終了]        │
├──────────────────────────────┤
│ 文字起こし                     │
│ ┌──────────────────────────┐ │
│ │ 発話テキスト...            │ │
│ └──────────────────────────┘ │
├──────────────────────────────┤
│ 観察ポイント                   │
│ - 相手の困り事...              │
│ - 実際に差し伸べた助け...      │
├──────────────────────────────┤
│ 良かった点                     │
│ - 親切をきっかけにした...      │
├──────────────────────────────┤
│ 短評素材 / AIメモ              │
│ - 短く整理されたコメント候補    │
└──────────────────────────────┘
```

SwiftUI的には、たぶんこう分けると作りやすい。

```text
MainView
├─ SessionToolbarView
├─ PartHeaderView
├─ PartControlsView
├─ TranscriptView
├─ ObservationPointsView
├─ GoodPointsView
└─ CommentMaterialView
```

特に慎平の要件だと、**1画面1パート表示**にするのは正解だと思う。幅を広げずに済むし、実演中に見る情報を絞れる。

Notion側のパート情報も、ガチガチに構造化しすぎず、アプリ内で必要な部分だけゆるく構造化する方針になっているので、画面側も「タイトル」「持ち時間」「観察メモ」「良かった点」「AIメモ」くらいに分けて扱えばよさそう [1]。

実装難易度はこんな感じ。

| 要素 | 難易度 |
|---|---:|
| セッション選択ドロップダウン | 低 |
| 新規作成・削除ボタン | 低 |
| パート表示切り替え | 低〜中 |
| 開始・終了ボタン | 低 |
| 経過時間表示 | 低 |
| 文字起こしテキストエリア | 中 |
| 観察ポイント表示 | 低 |
| 良かった点表示 | 低 |
| AIメモ表示 | 低 |
| Notion連携 | 中〜高 |
| 音声文字起こし | 中〜高 |

最初のSwiftUI実装では、**Notion連携や文字起こしは後回し**にして、まずはダミーデータで画面だけ作るのがいい。

例えば初期MVPの画面はこれで十分。

1. 上部：セッション操作
2. 中央：現在のパート情報
3. 操作：前へ・次へ・開始・終了・タイマー
4. 下部：文字起こし、観察ポイント、良かった点、短評素材

あと、AIメモはNotionの「🤖AIメモ」ブロック配下に書き込む想定で、既存ユーザーメモは上書きしない方針だから、画面上でも「AI生成エリア」と「ユーザーメモ」は混ぜないほうが安全 [1]。

なので、画面上の表示名は例えばこうすると分かりやすい。

```text
文字起こし
観察ポイント候補
良かった点候補
短評素材
Notion書き込み予定AIメモ
```
