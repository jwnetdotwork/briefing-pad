# AIエージェント向け macOS GUIアプリ テスト作成ガイドライン

## 基本方針

macOS GUIアプリでは、品質担保の中心をユニットテストだけに置かない。

GUIアプリの最終的な成果物は「画面」と「操作体験」であるため、ユニットテストだけでは本質的な正しさを検証しにくい。

AIエージェントは、テストコードを大量生成するのではなく、以下を優先すること。

1. 守る価値のある仕様を特定する
2. 保守コストの低いテストを選ぶ
3. 実装詳細ではなく振る舞いをテストする
4. GUIについては UI 操作テスト・手動確認・AIによる画面確認を重視する
5. 不要なユニットテストを増やさない

---

## テスト作成の原則

### 1. テストは多ければよいものではない

テストコードも保守対象である。

以下のようなテストは負債になりやすい。

- 実装詳細を固定するテスト
- Mock の戻り値を確認するだけのテスト
- 初期値を確認するだけのテスト
- UI内部構造に強く依存するテスト
- 頻繁に変わる文言やレイアウトを固定するテスト
- 仕様変更のたびに修正が必要になるテスト

AIエージェントは、テストを書く前に必ず「このテストは保守コストに見合う価値があるか」を判断すること。

---

### 2. 実装ではなく振る舞いをテストする

避けるべきテスト:

```text
- このメソッドが呼ばれること
- この内部 state が変更されること
- この private 関数が実行されること
- この ViewModel の特定プロパティに Mock の値が入ること
```

優先すべきテスト:

```text
- 無効な入力では保存できない
- 必須項目が空ならエラーを表示する
- 設定変更後に次回起動時も値が保持される
- 権限がない場合は操作できない
- ファイル読み込み失敗時に適切なエラーが表示される
```

---

### 3. GUIアプリではユニットテストの対象を絞る

macOS GUIアプリにおけるユニットテストは、主に以下に限定する。

- 純粋関数
- バリデーション
- データ変換
- 状態遷移
- 権限判定
- 設定値の保存・復元ロジック
- ファイルパス処理
- 日付・数値計算
- 複雑な条件分岐
- 過去バグの再発防止

UIそのものの正しさは、ユニットテストではなく UI テストや画面確認で担保する。

---

## テスト種別ごとの使い分け

## 1. Unit Test

### 目的

小さなロジックの壊れを検出する。

### 書くべき対象

- 入力と出力が明確な処理
- 仕様として安定している処理
- 境界値がある処理
- 影響範囲が大きい処理
- 過去にバグが出た処理

### 例

```text
- 空文字のタイトルは保存不可
- ファイルサイズが上限を超えた場合はエラー
- 無効なURLは登録不可
- 設定値が存在しない場合はデフォルト値を返す
- 日付範囲の開始日が終了日より後なら無効
```

### 避けるべき対象

```text
- View の初期 state 確認
- Mock が返した値を ViewModel が保持するだけの確認
- private メソッドの直接テスト
- UI コンポーネントの構造確認
- 単純な getter / setter の確認
```

---

## 2. Integration Test

### 目的

複数の部品が組み合わさったときに正しく動くかを確認する。

### 書くべき対象

- ファイル読み書き
- UserDefaults / Keychain / SQLite などの永続化
- API クライアントとデータ変換
- 設定変更と反映処理
- アプリ内サービス同士の連携
- エラー伝播

### 例

```text
- 設定画面で保存した値が次回起動時に復元される
- ファイルを読み込むと正しいモデルに変換される
- APIエラー時にユーザー向けエラーへ変換される
- キャッシュが存在する場合はAPIを呼ばずにデータを返す
```

---

## 3. UI Test / E2E Test

### 目的

ユーザー操作としてアプリが期待通りに動くかを確認する。

macOS GUIアプリでは重要度が高い。

### 書くべき対象

- 主要なユーザーフロー
- 起動から初期画面表示
- ファイル選択
- 設定変更
- フォーム入力
- 保存・削除・更新
- 検索
- エラー表示
- アクセシビリティ識別子を使った操作

### 例

```text
- アプリを起動するとメイン画面が表示される
- 必須項目を入力して保存できる
- 必須項目が空なら保存ボタンが無効になる
- 設定を変更して再起動しても値が維持される
- 存在しないファイルを開くとエラーダイアログが表示される
```

---

## 4. AIによる画面確認

### 目的

テストコードでは検出しにくい画面上の問題を確認する。

### 確認対象

- レイアウト崩れ
- ボタンや入力欄の視認性
- 操作導線の自然さ
- エラー表示の分かりやすさ
- 文言の違和感
- ダークモード対応
- ウィンドウサイズ変更時の崩れ
- キーボード操作
- アクセシビリティ

### AIエージェントへの指示例

```text
実際にアプリを起動し、主要なユーザーフローを操作してください。
以下を確認してください。

- 画面が崩れていないか
- 操作不能なボタンがないか
- 入力から保存まで自然に完了できるか
- エラー時にユーザーが理解できる表示になっているか
- ダークモードで視認性に問題がないか

問題があれば、スクリーンショットと再現手順を報告してください。
```

## ユニットテストを書くべき判断基準

以下のうち3つ以上に当てはまる場合、ユニットテストを書く価値がある。

```text
1. 入力と出力が明確である
2. 仕様として今後も残りそうである
3. 壊れたときの影響が大きい
4. 手動確認より自動テストの方が効率的である
5. 境界値や異常値が存在する
6. 複雑な条件分岐がある
7. 過去にバグが発生した箇所である
8. リファクタリング時の安全ネットになる
```

---

## ユニットテストを書かない判断基準

以下に当てはまる場合、原則としてユニットテストを書かない。

```text
1. 実装詳細を確認しているだけ
2. Mock の戻り値をなぞっているだけ
3. 初期値を確認しているだけ
4. UI の内部構造を固定している
5. 仕様変更で頻繁に壊れそう
6. 他のテストで十分担保されている
7. 失敗しても原因特定に役立たない
8. テストのために設計が不自然になる
```

## macOS GUIアプリで優先してテストすべき領域

### 高優先度

```text
- 保存処理
- 削除処理
- ファイル読み書き
- 設定の永続化
- 権限が関わる処理
- 課金・ライセンス認証
- データ変換
- エラー処理
- セキュリティに関わる処理
- 過去バグの再発防止
```

### 中優先度

```text
- 検索
- フィルタ
- ソート
- インポート / エクスポート
- 通知
- ショートカットキー
- ウィンドウ状態の復元
```

### 低優先度

```text
- 単純な画面表示
- 固定文言
- CSS / SwiftUI modifier / AppKit view 構造
- 単純なボタン表示
- 単純な初期 state
```

## AIが生成してはいけないテスト例

### 1. Mockを確認するだけのテスト

悪い例:

```swift
func testLoadUserSetsName() async {
    mockRepository.user = User(name: "Taro")

    await viewModel.load()

    XCTAssertEqual(viewModel.name, "Taro")
}
```

このテストは、Mock が返した値がそのまま入ったことを確認しているだけで、価値が低い。

許容されるのは、変換・分岐・エラー処理などの意味あるロジックが含まれる場合のみ。

---

### 2. 初期値だけを確認するテスト

悪い例:

```swift
func testInitialState() {
    let viewModel = SettingsViewModel()

    XCTAssertEqual(viewModel.title, "")
    XCTAssertFalse(viewModel.isLoading)
}
```

初期値が仕様として重要でない限り不要。

---

### 3. 実装詳細に依存するテスト

悪い例:

```swift
func testSaveCallsRepository() async {
    await viewModel.save()

    XCTAssertTrue(mockRepository.saveCalled)
}
```

呼び出し有無ではなく、ユーザーから見た結果を確認する。

改善例:

```swift
func testValidSettingsArePersisted() async {
    let store = InMemorySettingsStore()
    let service = SettingsService(store: store)

    await service.save(Settings(theme: .dark))

    XCTAssertEqual(await store.load().theme, .dark)
}
```

## UIテスト作成時のルール

### 1. accessibilityIdentifier を使う

UIテストでは、表示文言ではなく `accessibilityIdentifier` を優先する。

```swift
Button("Save") {
    save()
}
.accessibilityIdentifier("settings.saveButton")
```

理由:

- 文言変更に強い
- 多言語対応に強い
- テストの安定性が上がる

---

### 2. 主要導線に絞る

UIテストは壊れやすく実行時間も長い。

そのため、すべての画面を網羅しない。

優先する導線:

```text
- 起動できる
- 主要画面が開く
- データを作成できる
- データを編集できる
- データを削除できる
- 設定を変更できる
- エラー時に回復できる
```

---

### 3. 細かい見た目を固定しない

避けるべき検証:

```text
- ボタンの位置が x=100 y=200
- 余白が 16px
- Text が何番目の Stack にある
- SwiftUI の View 階層
```

必要な場合のみ、スクリーンショットテストや AI 画面確認で補う。

## テスト作成フロー

AIエージェントは、以下の順番で進めること。

## Step 1. 仕様を整理する

最初にコードを書かない。

```text
- この機能の目的は何か
- ユーザーにとって重要な振る舞いは何か
- 壊れると困る処理は何か
- 入力と出力が明確なロジックはどこか
- UIで確認すべき内容は何か
```

---

## Step 2. テスト候補を分類する

以下に分類する。

```text
- Unit Test で確認するもの
- Integration Test で確認するもの
- UI Test で確認するもの
- AI/人間の画面確認で確認するもの
- テストしないもの
```

---

## Step 3. 不要なテストを除外する

以下は除外する。

```text
- 初期値確認だけのテスト
- Mockをなぞるだけのテスト
- 実装詳細固定のテスト
- 重複テスト
- 仕様変更に弱すぎるテスト
```

---

## Step 4. 最小限のテストだけ実装する

最初から網羅しない。

目安:

```text
- Unit Test: 重要なロジックのみ
- Integration Test: 永続化・ファイル・API境界
- UI Test: 主要導線のみ
- AI画面確認: 見た目・操作感・崩れ
```

---

## Step 5. テストをレビューする

テスト実装後、以下を確認する。

```text
- このテストは何を保証しているか
- その保証に価値があるか
- 実装詳細に依存していないか
- 仕様変更時に過剰な負担にならないか
- 同じ内容を別のテストで確認していないか
- 失敗時に原因が分かりやすいか
```

---

## AIエージェントへの標準指示文

```text
macOS GUIアプリのテストを作成してください。

ただし、テストコードをすぐに書かないでください。
まず以下を整理してください。

1. この機能で守るべき仕様
2. Unit Test で確認すべきもの
3. Integration Test で確認すべきもの
4. UI Test で確認すべきもの
5. AIまたは人間の画面確認で確認すべきもの
6. テストしない方がよいもの

方針:
- テストは多ければよいとは考えない
- 実装詳細ではなく振る舞いをテストする
- Mock の戻り値を確認するだけのテストは禁止
- 初期値確認だけのテストは禁止
- private メソッドの直接テストは禁止
- UIの細かい構造や文言を固定するテストは避ける
- 境界値、異常値、データ変換、保存処理、権限、エラー処理を優先する
- GUIについては UI Test と実際の画面確認を重視する
- 保守コストが価値を上回るテストは作成しない

整理したテスト方針を提示し、承認されたテストだけ実装してください。
```

```text

- [ ] 実装詳細ではなく振る舞いを確認している
- [ ] Mockをなぞるだけのテストがない
- [ ] 初期値確認だけのテストがない
- [ ] UI構造に過度に依存していない
- [ ] 主要なユーザーフローが確認されている
- [ ] 保存・削除・ファイル操作など重要処理が確認されている
- [ ] 過去バグの再発防止テストがある
- [ ] 不要なテストを追加していない
- [ ] テストが失敗したとき原因を特定しやすい
```

## まとめ

macOS GUIアプリにおけるテスト戦略は、以下を基本とする。

```text
ユニットテストは重要なロジックに絞る。
統合テストで永続化・ファイル・API境界を確認する。
UIテストで主要なユーザーフローを確認する。
画面の正しさや操作感は AI または人間が実際に確認する。
AIにテストを丸投げせず、書くべきテストを選別する。
```

最も重要なのは、  
**AIにテストを大量生成させることではなく、価値のあるテストだけを選んで作らせること。**
